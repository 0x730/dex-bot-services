# Public GitHub Release Technical Spec

## Goal

Publish a public GitHub repository from a private source repository without
copying private Git history, local secrets, IDE state, or generated logs.

## Source Of Truth

- Private development remains in the current Bitbucket repository.
- Public GitHub receives a clean snapshot generated from the working tree.
- Release procedure and acceptance checks live in this repository and must be
  reusable for other projects.

## Contract

- Public release snapshots are generated by `npm run release:public:prepare`.
- The export must not copy `.git`, `.env`, `.env.*`, IDE files, `node_modules`,
  generated logs, `archive/`, root `PM/`, `0x730/PM/`, or legacy todo notes.
- The export must not copy local/live smoke scripts or stale tool configs that
  are not part of the reproducible public test contract.
- `.env.example` and `.env.release.example` are the only environment files
  allowed in a public snapshot.
- The export initializes a new Git repository with no private source history.
- `npm run release:public:publish` must publish the snapshot, tag, and GitHub
  release from a local gitignored release env file.
- The one-command publisher must use `GITHUB_PAT` directly: GitHub REST API for
  repository/release operations and plain `git` for source/tag pushes.
- When `RELEASE_VERSION=auto`, the publish script must query GitHub
  releases/tags and publish the next patch version.
- A concrete `RELEASE_VERSION` must override automatic versioning.
- `RELEASE_REPLACE_MAIN=true` must be required before replacing a non-empty
  remote `main`, and replacement must use `--force-with-lease`.
- Before using `--force-with-lease`, the publisher must fetch `origin/main` into
  the generated snapshot repository so the lease is based on the current remote
  branch tip.
- The script prints the exact follow-up commands for creating a public GitHub
  repository, pushing the initial commit, tagging a version, and creating a
  GitHub release.
- The script must run `npm test` and `npm run check:public` before export unless
  explicitly called with `--skip-verify`.
- The script must block publication when required public metadata is missing:
  `LICENSE`, non-generic `package.json` name, description, and matching license.
- The script must block known placeholder public contact links for fake support
  email, Discord, or Telegram channels.
- The script must require `--secrets-rotated` before creating a snapshot.
- The script must support `--dry-run` for checks that do not create a snapshot
  and do not require the secrets-rotation acknowledgement.
- The human-facing runbook lives in `0x730/PM/runbooks/public-github-release.md`.

## Command

Prepare only:

```bash
npm run release:public:prepare -- \
  --out /tmp/bots-public \
  --version 1.0.0 \
  --github-repo 0x730/dex-bot-services \
  --secrets-rotated \
  --force
```

One-command publish:

```bash
npm run release:public:publish
```

## Acceptance Criteria

- Running the command creates a public snapshot directory.
- The snapshot does not contain private Git history.
- The snapshot does not contain real env files or IDE files.
- `package.json` in the snapshot uses the requested `--version`.
- Missing public release metadata fails before a snapshot is created.
- Missing `--secrets-rotated` fails before a snapshot is created.
- `--dry-run` runs release checks without creating a snapshot.
- `npm run release:public:publish -- --dry-run` prints the publish operations
  without pushing to GitHub.
- Automatic release versioning starts at `0.0.1` when no GitHub semver
  releases/tags exist.
- Existing remote `main` is not replaced unless `RELEASE_REPLACE_MAIN=true`.
- Existing remote `main` replacement fetches `origin/main` first, then pushes with
  `--force-with-lease`.
- The printed commands can create and release a new public GitHub repository from
  the snapshot.

## Pre-Release Human Checklist

- Rotate any secret that was ever committed in the private repository.
- Choose and add the correct public license/copyright holder.
- Choose the public package/repository name and description.
- Review `README.md`, docs, and `0x730/spec/` for business context that should
  not be public.
- Run a smoke check for production-used services before tagging.
